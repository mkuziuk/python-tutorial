## Что это

Аннотации типов описывают, какие значения ожидает код и что он возвращает.
Python обычно не проверяет аннотации при запуске, но редактор, анализатор типов
и читатель получают явный контракт.

Начнём с функции, которая переводит целое число баллов в текстовую метку:

```python
def risk_label(score: int) -> str:
    """Возвращает метку для целого числа баллов."""
    return "высокий риск" if score >= 7 else "низкий риск"


print(risk_label(8))
```

```text
высокий риск
```

`score: int` описывает параметр, а `-> str` — результат. Вывод подтверждает
поведение функции для значения `8`, но не доказывает, что Python отклонит
строку: аннотация сама по себе не выполняет преобразование и не заменяет
валидацию.

Запись `-> None` используют для функции, которая выполняет действие и не
возвращает полезного значения. Мы описали простые значения; следующий этап —
указать форму контейнеров, передаваемых между функциями.

## Типы коллекций

Тип элемента записывается в квадратных скобках:

```python
names: list[str] = ["Алина", "Никита"]
scores: dict[str, int] = {"Алина": 7}
pair: tuple[str, str] = ("report-a", "report-b")

print(names[0], scores["Алина"], pair)
```

```text
Алина 7 ('report-a', 'report-b')
```

Контейнеры имеют разные контракты: `names` — список строк, `scores` —
отображение `имя → целое число`, а `pair` — кортеж ровно из двух строк.
Фактический вывод показывает значения, но соответствие аннотациям обычно
проверяет отдельный анализатор типов.

Для кортежа произвольной длины указывают тип элемента и многоточие. Сначала
определим используемый тип, чтобы пример был самодостаточным:

```python
from dataclasses import dataclass


@dataclass
class Evidence:
    title: str


events: tuple[Evidence, ...] = (
    Evidence("Письмо"),
    Evidence("Журнал входов"),
)
print(len(events), events[0].title)
```

```text
2 Письмо
```

`tuple[Evidence, ...]` означает кортеж любой длины, каждый элемент которого —
`Evidence`. Число `2` подтверждает длину конкретного примера, а не ограничение
типа. Мы получили типизированную внутреннюю модель. Теперь нужно провести к
ней внешние JSON-данные, значения которых заранее неоднородны.

## `Any` на границе внешних данных

JSON-объект после чтения представлен словарём со значениями разных типов.
`Any` временно обозначает, что анализатор не знает конкретный тип значения:

```python
from typing import Any


def load_record(data: dict[str, Any]) -> Evidence:
    """Преобразует внешний словарь в объект с известным полем title."""
    return Evidence(title=str(data["title"]))


item = load_record({"title": "Письмо", "priority": 3})
print(item)
```

```text
Evidence(title='Письмо')
```

На входе `data` допускает неоднородные значения, но функция сразу создаёт
`Evidence`, внутри которого `title` имеет известный тип `str`. Ключ `priority`
в этом учебном преобразовании не используется, а отсутствие обязательного
`title` приведёт к `KeyError`. `Any` уместен на короткой границе чтения, но
внутри остальной программы он скрыл бы полезные проверки.

## `None` и объединения

Запись `str | None` означает «строка или отсутствие значения». До строковой
операции нужно отделить `None`:

```python
def normalized_note(note: str | None) -> str:
    """Возвращает очищенную заметку или явную замену для отсутствия."""
    if note is None:
        return "нет заметки"
    return note.strip()


print(normalized_note(None))
print(normalized_note("  проверено  "))
```

```text
нет заметки
проверено
```

Первая строка подтверждает отдельную ветвь отсутствующего значения, вторая —
очистку строки. После проверки `note is None` анализатор знает, что в
оставшейся ветви `note` имеет тип `str`, поэтому вызов `strip()` безопасен.

Мы отделили необязательное значение на границе функции. Для фабричного метода
класса остаётся точно описать тип создаваемого объекта.

## `Self` в фабричном методе

`Self` означает объект текущего класса. Ниже приведён полный пример со всеми
предварительными определениями:

```python
from dataclasses import dataclass
from typing import Any, Self


@dataclass
class Evidence:
    title: str

    @classmethod
    def from_dict(cls, data: dict[str, Any]) -> Self:
        """Создаёт объект текущего класса из внешнего словаря."""
        return cls(title=str(data["title"]))


item = Evidence.from_dict({"title": "Письмо"})
print(type(item).__name__, item.title)
```

```text
Evidence Письмо
```

`cls(...)` создаёт объект класса, на котором вызван метод, а `Self` сохраняет
этот тип в контракте результата. Поэтому возможный подкласс получит более
точную аннотацию, чем при записи `-> Evidence`. Вывод проверяет фактический
класс и значение поля для базового примера; корректность всех входных словарей
он не гарантирует.

## Как проверить аннотации

Обычный запуск проверяет поведение программы:

```bash
python example.py
```

Статический анализатор типов выполняют отдельно. Если в проекте уже настроен
`mypy`, команда выглядит так:

```bash
python -m mypy example.py
```

```text
Success: no issues found in 1 source file
```

Этот результат означает, что анализатор не нашёл противоречий в доступных ему
аннотациях. Он не проверяет смысл порога риска, наличие файлов или все
возможные данные во время выполнения. После изменения контракта полезно
повторить оба шага: запустить программу и затем анализатор, настроенный в
конкретном проекте.

## Типичные ловушки

- Аннотация не превращает значение автоматически:
  `score: int = "7"` по-прежнему сохраняет строку.
- Не используйте `Any` внутри всей программы только потому, что входной JSON
  неоднороден; преобразуйте внешние данные в известную модель на границе.
- Тип `list[str]` не принимает `None`, если это не указано явно.
- Аннотации должны прояснять границы функций и моделей, а не дублировать
  очевидную локальную переменную на каждой строке.
- Успешный статический анализ не заменяет запуск программы и проверки данных.

## Где встречается в расследованиях

В текущей первой арке подробные аннотации типов не входят в финальное
обязательное задание. Страница остаётся справкой для расширения функций курса:
начинайте с параметров и результата функции, а `Any` оставляйте только у
границы внешних данных.
