## Когда строка не независима

Несколько строк могут относиться к одному человеку, документу, устройству, району или исходному изображению. Если связанные объекты попадают и в обучающую, и в тестовую выборку, модель получает подсказку о тестовом объекте.

Например, `source-A` породил два снимка: `A-original` и `A-noisy`. Построчное разбиение может отправить первый снимок в обучающую выборку, второй — в тестовую и формально не повторить `sample_id`. Но независимый объект здесь — исходник A, поэтому связь всё равно пересекла границу. Групповое разбиение держит оба снимка вместе: они попадают либо в обучение, либо в проверку.

| sample_id | source_id | Небезопасное построчное разбиение |
| --- | --- | --- |
| A-original | A | train |
| A-noisy | A | test |
| B-original | B | train |

```python
train_source_ids = ["A", "B"]
test_source_ids = ["C", "D"]
overlap = set(train_source_ids) & set(test_source_ids)
print(sorted(overlap))
assert not overlap
```

```text
[]
```

Пустой список подтверждает, что ни один `source_id` не пересёк границу. Такая
проверка должна выполняться на фактическом разбиении, а не только на примере.
Единицу независимости определяет задача будущего применения. Для вариантов изображения в II-06 это `source_id`, а не `sample_id`.

## StratifiedGroupKFold

Проверим разбиение на десяти источниках. Каждый источник даёт две строки, а
`source_id` должен целиком оставаться в одном фолде. Небольшой конвейер нужен
здесь не для сравнения алгоритмов, а чтобы получить по одной macro-F1 на фолд:

```python
import numpy as np
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import StratifiedGroupKFold, cross_validate
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler

source_id = np.repeat([f"S{index}" for index in range(10)], 2)
y = np.repeat([0, 1] * 5, 2)
X = np.column_stack([
    y + np.tile([-0.15, 0.15], 10),
    np.arange(20) % 3,
])

pipeline = make_pipeline(
    StandardScaler(),
    LogisticRegression(max_iter=1000, random_state=42),
)

cv = StratifiedGroupKFold(n_splits=5, shuffle=True, random_state=42)
scores = cross_validate(
    pipeline,
    X,
    y,
    # groups задаёт принадлежность к частям; модель обучается на X.
    groups=source_id,
    cv=cv,
    scoring="f1_macro",
    n_jobs=1,
)

for train_index, test_index in cv.split(X, y, groups=source_id):
    train_groups = set(source_id[train_index])
    test_groups = set(source_id[test_index])
    assert train_groups.isdisjoint(test_groups)

print(np.round(scores["test_score"], 3))
print(
    f"mean={scores['test_score'].mean():.3f}; "
    f"groups={np.unique(source_id).size}"
)
print("group overlap: 0")
```

```text
[1. 1. 1. 1. 1.]
mean=1.000; groups=10
group overlap: 0
```

Разбиение старается сохранять доли классов и одновременно держит каждую группу целиком в одном фолде. После построения фолдов полезно программно проверить отсутствие пересечений.

`scores["test_score"]` содержит пять macro-F1, по одному на фолд. Macro-F1
лежит от 0 до 1, больше — лучше, а каждый класс имеет одинаковый вес. В этом
искусственном легко разделимом наборе все пять значений равны `1.0`; это
проверяет механику примера, но не обещает такого качества на реальных
изображениях. Последняя строка подтверждает отсутствие пересечения групп, а
`groups=10` показывает число независимых источников: одинаковое число строк
при малом числе групп не делает оценку точнее.

## Сдвиг распределения

**Сдвиг распределения (distribution shift)** означает, что данные применения отличаются от обучающих: другая партия сканера, регион, период или процесс отбора. Это не обязательно изменение каждого числа — достаточно, чтобы иначе встречались классы, шум или условия измерения. Обычная IID-оценка предполагает, что независимые обучающие и тестовые примеры получены примерно одним процессом; она не гарантирует качество при сдвиге.

Изолированная проверка работает так:

1. заранее определите срез, соответствующий условиям применения;
2. не используйте его для признаков, модели, параметров и порога;
3. зафиксируйте всю процедуру;
4. откройте набор один раз;
5. сообщите результат вместе с происхождением и ограничениями.

Падение показывает чувствительность к конкретному сдвигу, но не объясняет автоматически причину и не покрывает все будущие сдвиги.

Например, падение macro-F1 с `0.84` на обычной проверке до `0.68` на новой
партии равно `0.16` по шкале 0–1, или 16 процентным пунктам. Это наблюдение относится только к
заранее изолированной партии и не доказывает, что причиной был сканер. Мы
зафиксировали величину сдвига качества; следующий шаг — разобрать ошибки по
классам и условиям получения данных, не переобучаясь на этой партии.

## Бутстреп по независимой единице

Чтобы оценить разброс метрики, бутстреп повторно выбирает независимые единицы с возвращением. Базовые значения слов «разброс» и «интервал» есть в [математическом словаре](../numpy/#math-vocabulary). Если у источника несколько вариантов, пересэмплируйте `source_id` и берите все его строки вместе. Построчный бутстреп снова ошибочно считал бы связанные варианты независимыми.

«С возвращением» означает, что выбранная единица может появиться снова, а другая — не попасть в повтор. Для источников `[A, B, C, D]` один бутстреп-повтор может быть `[A, C, C, B]`. Если у C три строки, все три включаются дважды. После сотен таких повторов берут, например, 2,5-й и 97,5-й процентили метрики как границы 95%-го интервала.

Интервал отражает вариативность в рамках набора и схемы ресемплинга. Это не гарантия будущего качества и не исправление смещения данных.

## Карточка модели

Короткая карточка модели должна отвечать минимум на вопросы:

- для какой задачи и пользователя создана система;
- какие применения допустимы и недопустимы;
- откуда данные и какова единица наблюдения;
- как разделены обучающая, проверочная и тестовая выборки и группы;
- какая модель и версия зависимостей использованы;
- какие метрики, пороги, срезы и интервалы получены;
- какие ограничения и известные режимы отказа остались;
- что нужно проверить перед внедрением.

Карточка не делает модель безопасной сама по себе. Она не даёт спрятать важные решения за одним значением метрики.

## Типичные ловушки

- Группировать по уникальному `sample_id`, когда связь задаёт `source_id`.
- Подбирать модель по изолированной партии данных.
- Считать синтетический стресс-тест точной копией реального устройства.
- Выполнять бутстреп по строкам при кластерных данных.
- Публиковать карточку модели без версии данных и схемы разбиения.
