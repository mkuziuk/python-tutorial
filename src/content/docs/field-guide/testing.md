## Что такое тест

Модульный тест вызывает небольшой фрагмент программы с известными входными
данными и сравнивает наблюдаемый результат с ожидаемым. Тест не доказывает
отсутствие всех ошибок, но быстро сообщает, сломалось ли уже проверенное
поведение.

Разберём цикл на функции, которая строит соседние пары слов. Сначала нужен
проверяемый код. Сохраните его в файле `copy_paste_detector.py`:

```python
def make_ngrams(words: list[str], size: int) -> list[tuple[str, ...]]:
    """Возвращает все соседние группы из size слов."""
    if size <= 0:
        raise ValueError("size должен быть положительным")

    # Каждый срез содержит одну соседнюю группу заданной длины.
    return [
        tuple(words[start : start + size])
        for start in range(len(words) - size + 1)
    ]
```

Функция получает список строк и возвращает список кортежей. Для трёх слов и
`size=2` возможны две начальные позиции, поэтому ожидаемая форма результата —
список из двух пар. Теперь это правило можно зафиксировать отдельным тестом.

## Первая проверка

Создайте рядом файл `test_copy_paste_detector.py`. Метод теста начинается с
`test_`, чтобы `unittest` нашёл его автоматически:

```python
import unittest

from copy_paste_detector import make_ngrams


class NgramTests(unittest.TestCase):
    def test_builds_pairs(self):
        # Небольшой вход позволяет записать ожидаемые пары буквально.
        result = make_ngrams(["ночной", "сигнал", "архива"], size=2)
        self.assertEqual(
            result,
            [("ночной", "сигнал"), ("сигнал", "архива")],
        )


if __name__ == "__main__":
    unittest.main()
```

`result` хранит фактический список пар, а `assertEqual` сравнивает его со
списком, который следует из правила задачи. Один тест лучше посвящать одному
наблюдаемому правилу: тогда имя упавшего метода сразу указывает, какое
поведение нарушено.

## Как запускать

Откройте в системном файловом приложении папку с двумя файлами. Щёлкните по
папке правой кнопкой мыши и выберите команду открытия терминала в этой папке
(название команды зависит от системы). Так терминал сразу начнёт работу в
нужной папке и вводить `cd` не потребуется.

Запустите тестовый файл:

```bash
python -m unittest test_copy_paste_detector -v
```

```text
test_builds_pairs (test_copy_paste_detector.NgramTests.test_builds_pairs) ... ok

----------------------------------------------------------------------
Ran 1 test in 0.000s

OK
```

Строка `ok` подтверждает проверенное правило только для указанного примера:
функция построила две соседние пары. Она ещё не проверяет недопустимый размер
или другие граничные случаи, поэтому следующим этапом добавим отдельное
ожидаемое исключение.

Если тесты проекта действительно лежат в папке `tests/`, весь комплект можно
запустить командой:

```bash
python -m unittest discover -s tests -v
```

Чтобы сосредоточиться на одном методе, укажите полный путь импорта:

```bash
python -m unittest tests.test_copy_paste_detector.CopyPasteDetectorTests.test_make_ngrams_returns_word_tuples -v
```

Эти команды — справочные шаблоны. Используйте имя модуля и класса из своего
проекта.

## Проверяем границу входных данных

Требование `size > 0` относится к границе функции. Добавьте в `NgramTests`
второй метод:

```python
def test_rejects_non_positive_size(self):
    with self.assertRaises(ValueError):
        make_ngrams(["текст"], size=0)
```

Контекст `assertRaises` считает тест успешным только при `ValueError`. После
повторного запуска вывод должен содержать два результата:

```text
test_builds_pairs (...) ... ok
test_rejects_non_positive_size (...) ... ok

----------------------------------------------------------------------
Ran 2 tests in 0.000s

OK
```

Две строки `ok` подтверждают разные свойства: обычный вход даёт пары, а
недопустимый размер отклоняется. Мы проверили результат и одну границу. Для
кода, который пишет файлы, требуется ещё изолировать побочный эффект от
рабочих данных.

## Проверяем работу с файлами

`TemporaryDirectory` создаёт отдельную временную папку и удаляет её после
выхода из блока. Такой тест не перезапишет учебные входные файлы:

```python
import tempfile
from pathlib import Path


with tempfile.TemporaryDirectory() as tmp:
    output = Path(tmp) / "report.json"
    output.write_text("{}", encoding="utf-8")

    print(output.exists(), output.read_text(encoding="utf-8"))
```

```text
True {}
```

`True` подтверждает создание файла, а `{}` — его точное содержимое. В классе
`unittest.TestCase` эти же свойства обычно проверяют через
`self.assertTrue(output.exists())` и
`self.assertEqual(output.read_text(encoding="utf-8"), "{}")`.

## Как читать сбой

- `OK` — все запущенные проверки прошли.
- `FAIL` — тест получил результат, но проверка сравнения не прошла. Ниже
  показаны фактическое и ожидаемое значения.
- `ERROR` — код не смог завершить тест: например, импорт не нашёл функцию или
  программа выбросила неожиданное исключение.

Читайте traceback снизу вверх: сначала найдите последнюю строку с типом ошибки,
затем первую строку, которая относится к вашему файлу. После исправления снова
запустите сначала упавший тест, а затем весь доступный комплект: успешная
отдельная проверка не гарантирует, что соседнее поведение осталось целым.

## Полезные проверки

```python
self.assertEqual(actual, expected)
self.assertAlmostEqual(score, 0.617)
self.assertIn("доступ", tags)
self.assertTrue(path.exists())
```

Выбирайте проверку по наблюдаемому контракту функции: точное значение,
приближённое число, наличие элемента или логическое условие.

## Типичные ловушки

- Не меняйте ожидаемый результат только ради зелёного теста; сначала проверьте
  правило задачи.
- Тест не должен зависеть от порядка запуска других тестов.
- Не записывайте результаты теста поверх учебных данных.
- Проверяйте результат, важный пользователю или следующей функции, а не
  случайные внутренние детали реализации.
- Не считайте один удачный пример доказательством корректности для всех
  возможных входов.

## Где встречается в расследованиях

В Части I выполняйте промежуточные команды проверки, приведённые в самих
главах. Служебные тесты репозитория предназначены для сопровождающих курса и
не входят в студенческие архивы. Эта страница остаётся справкой: по ней можно
добавить собственные маленькие тесты к функции, если вы хотите проверить
решение на дополнительных примерах.
