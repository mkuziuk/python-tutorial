## Что это

Jupyter Notebook соединяет выполняемые ячейки Python, текст, таблицы и графики в одном файле `.ipynb`. Это удобно для расследования: рядом с кодом остается вопрос, наблюдение и интерпретация. Но kernel хранит состояние между запусками, поэтому notebook легко сделать невоспроизводимым.

## Главная модель состояния

Когда вы выполняете ячейку, имена попадают в память kernel. Порядок на странице и порядок выполнения могут различаться:

```python
# Ячейка 1
threshold = 0.70

# Ячейка 2
labels = probabilities >= threshold
```

Если сначала выполнить ячейку 2 после старого `threshold = 0.50`, результат не соответствует видимому коду. Номера `In [ ]` помогают заметить нарушение, но надежная проверка одна: **Restart Kernel and Run All**.

## Локальный запуск

После распаковки ZIP создайте окружение в папке дела:

```bash
python3 -m venv .venv
source .venv/bin/activate        # macOS / Linux
python -m pip install -r requirements.txt
jupyter lab
```

В Windows PowerShell активация выглядит так:

```powershell
py -3 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
jupyter lab
```

Открывайте notebook из корня дела. Тогда относительный путь `data/titanic.csv` одинаков локально и в Colab.

## Запуск в Colab

Ссылка из брифинга открывает канонический learner notebook на GitHub. Первая ячейка:

1. определяет среду;
2. загружает соответствующий ZIP;
3. проверяет SHA-256;
4. распаковывает ту же структуру папок;
5. выставляет рабочий каталог.

Выполните bootstrap один раз, затем **Runtime → Run all**. Среда Colab временная: после сброса runtime загруженные файлы исчезают, поэтому сохраняйте заполненный `.ipynb`, а не рассчитывайте на диск сессии.

## Хороший учебный notebook

- одна ячейка делает один логический шаг;
- импорты и `RANDOM_STATE` объявлены в начале;
- данные не скачиваются из меняющегося источника во время анализа;
- график следует сразу за расчетом и имеет подписи;
- Markdown объясняет наблюдение, но не выдает догадку за факт;
- последняя ячейка содержит структурированное аудиторское мемо.

## Типичные ловушки

- Выполнять ячейки снизу вверх и забывать о старых переменных.
- Исправить функцию, но не перезапустить ячейки, которые ее вызывали.
- Записать абсолютный путь вроде `/Users/name/data.csv`.
- Установить пакет без фиксированной версии и получить другой результат через месяц.
- Очистить код, но оставить output от предыдущей версии.

Перед сдачей перезапустите kernel, выполните все ячейки сверху вниз и убедитесь, что нет ручных действий вне notebook.
