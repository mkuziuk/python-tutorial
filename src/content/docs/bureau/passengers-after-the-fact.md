## Сюжет

После вступительного экзамена Тимур приносит первый реальный пакет «Компаса». В отчёте о выживаемости пассажиров заявлена почти безошибочная классификация. Среди столбцов есть `boat` и `body`. Антон объясняет, что платформа «использует все доступные сигналы».

Доступность признака определяется моментом, когда модель должна сделать прогноз. Номер спасательной шлюпки и номер найденного тела стали известны **после** исхода и поэтому создают утечку. Ваша задача — формально зафиксировать эту утечку и показать её влияние на оценку.

## Что расследуем

1. Прочитайте карточку происхождения snapshot и проверьте контрольную сумму.
2. Исследуйте типы, пропуски, уникальные значения и связь столбцов с целевой переменной.
3. Для каждого кандидата в признаки отметьте: доступен до события, появляется после события или неоднозначен.
4. Уберите `boat` и `body` из корректного набора признаков и зафиксируйте внешнюю стратифицированную тестовую выборку.
5. Для чисел соедините `SimpleImputer` и масштабирование; для категорий — `SimpleImputer` и `OneHotEncoder`.
6. Объедините ветки в `ColumnTransformer`, затем добавьте логистическую регрессию в `Pipeline`.
7. На одной и той же тестовой выборке сравните корректный конвейер и учебную версию с утечкой.

Главный результат — воспроизводимое измерение того, насколько оценка меняется после удаления недоступной информации.

## Код расследования

Тетрадь содержит завершённые направляемые листинги. Сначала проверьте локальный файл и создайте `DataFrame`; `DATA_DIR` и `sha256_file()` создаются в начальной ячейке:

```python
DATASET_SHA256 = "c617db2c7470716250f6f001be51304c76bcc8815527ab8bae734bdca0735737"
data_path = DATA_DIR / "titanic.csv"
# Контрольная сумма связывает выводы с точной версией локального снимка.
actual_sha256 = sha256_file(data_path)
if actual_sha256 != DATASET_SHA256:
    raise RuntimeError("Контрольная сумма titanic.csv не совпала")

passengers = pd.read_csv(data_path, na_values=["?"])
passengers["survived"] = passengers["survived"].astype(int)
display(passengers.head(3))
print("Форма DataFrame:", passengers.shape)
```

Форма `(1309, 14)` подтверждает ожидаемый snapshot, а строки `head(3)` показывают единицу наблюдения — одного пассажира. Ранний audit дополнительно показывает типы, пропуски, уникальные значения, числовые диапазоны, полные дубликаты и распределение `survived`; затем внешний test замораживается до визуальной разведки и выбора сценариев. После этого готовая карта доступности фиксирует не тип pandas, а момент появления поля:

```python
availability_rows = [
    ("pclass", "available", "класс билета известен до исхода"),
    ("survived", "target", "факт выживания — предсказываемый исход"),
    ("name", "excluded_identifier", "прямой идентификатор и свободный текст"),
    ("sex", "available", "записан до исхода"),
    ("age", "available", "мог быть известен до исхода"),
    ("sibsp", "available", "состав семьи в манифесте"),
    ("parch", "available", "состав семьи в манифесте"),
    ("ticket", "excluded_identifier", "высокая кардинальность; не нужен baseline"),
    ("fare", "available", "стоимость билета известна заранее"),
    ("cabin", "excluded_identifier", "разреженный код; исключён из baseline"),
    ("embarked", "available", "порт посадки известен заранее"),
    ("boat", "post_outcome", "номер шлюпки фиксирует результат эвакуации"),
    ("body", "post_outcome", "номер найденного тела появляется после гибели"),
    ("home.dest", "excluded_identifier", "свободный текст высокой кардинальности"),
]
field_audit = pd.DataFrame(
    availability_rows, columns=["feature", "decision_status", "reason"]
)

```

Карта устанавливает семь доступных признаков и отделяет `boat` и `body` как `post_outcome`. Отдельная группировка показывает, что заполненность этих полей резко различается между классами; это описательная связь, а не причинное доказательство. После аудита тетрадь явно разделяет два набора:

```python
honest_numeric = ["age", "sibsp", "parch", "fare"]
honest_categorical = ["pclass", "sex", "embarked"]
honest_features = honest_numeric + honest_categorical
post_outcome_features = ["boat", "body"]

X_train_honest = train_data[honest_features]
print("Честные признаки:", honest_features)
```

`X_train_honest` имеет семь столбцов в зафиксированном порядке. Тестовая матрица пока не создаётся: тестовые строки не участвуют в выборе признаков, обучении медиан, категорий, масштабирования или коэффициентов.

Обучаемые преобразования собраны вместе с моделью. Поэтому вызов `.fit(...)` ниже передаёт импьютеру, масштабированию и кодировщику только обучающие данные:

```python
def build_logistic_pipeline(
    numeric_features: list[str], categorical_features: list[str]
) -> Pipeline:
    numeric_steps = Pipeline(
        [
            ("imputer", SimpleImputer(strategy="median", add_indicator=True)),
            ("scaler", StandardScaler()),
        ]
    )
    categorical_steps = Pipeline(
        [
            ("imputer", SimpleImputer(strategy="constant", fill_value="__MISSING__")),
            ("onehot", OneHotEncoder(handle_unknown="ignore")),
        ]
    )
    preprocessing = ColumnTransformer(
        [
            ("numeric", numeric_steps, numeric_features),
            ("categorical", categorical_steps, categorical_features),
        ]
    )
    # Pipeline обучает преобразования на обучающей части каждого разбиения.
    return Pipeline(
        [
            ("preprocess", preprocessing),
            (
                "model",
                LogisticRegression(
                    solver="liblinear", max_iter=1000, random_state=RANDOM_STATE
                ),
            ),
        ]
    )
```

Перед числами зафиксируем смысл метрик. **Accuracy** — доля правильных классов; больше лучше, но при балансе около 62/38 значение около 0,62 доступно модели, всегда выбирающей большинство. **ROC AUC** — вероятность того, что случайно выбранный выживший получит более высокий score, чем случайно выбранный невыживший. `0.5` соответствует случайному ранжированию, `1.0` — идеальному; больше лучше. ROC AUC не является долей правильных ответов и не доказывает калибровку вероятностей.

Dummy, честный конвейер и контролируемый антипример с `boat` и `body` сначала полностью обучаются на одних и тех же train-строках. Архитектуры, списки признаков и стандартный порог `0.5` фиксируются до просмотра test.

Только после этого одна итоговая ячейка впервые создаёт тестовые матрицы, получает прогнозы всех трёх сценариев и вычисляет для них accuracy и ROC AUC одним набором функций. Базовый ROC AUC не записывается литералом: `roc_auc_score` получает постоянный score Dummy и возвращает `0.5`.

Честная модель превосходит Dummy, но одна внешняя оценка ещё не означает готовность к внедрению. Резкий рост accuracy и ROC AUC после добавления `boat` и `body` измеряет эффект сведений после исхода; строка `deployable=False` запрещает трактовать антипример как кандидат на использование.

Финальная таблица группирует ошибки честной модели по `sex × pclass` и выводит `n`, `errors`, `error_rate`. Первая строка после сортировки — конкретный наблюдаемый режим с наибольшей долей ошибок. Интерпретировать её нужно вместе с `n`: малая подгруппа даёт нестабильную долю и не поддерживает причинный вывод.
