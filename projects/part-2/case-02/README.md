# Расследование II-02. Пассажиры после факта

Поставщик показывает почти безошибочный прогноз выживания пассажиров «Титаника». Бюро должно выяснить, какие поля действительно существовали в момент прогноза, а какие появились только после исхода.

## Результат расследования

За 4–5 часов вы:

- исследуете типы, диапазоны, дубликаты, распределение цели, пропуски и момент доступности полей;
- распознаете `boat` и `body` как утечку сведений о результате;
- один раз зафиксируете внешнюю стратифицированную тестовую выборку;
- соберёте `ColumnTransformer` с заполнением пропусков и кодированием категорий через `OneHotEncoder`;
- сначала зафиксируете три сценария на train, затем одним блоком вычислите для них accuracy и ROC AUC на внешнем test;
- разберёте таблицу ошибок по `sex × pclass` с учётом размера подгрупп;
- сформулируете границы доказанного.

## Подготовка

Нужен Python 3.12 или 3.13.

```bash
python3 -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
python -m pip install -r requirements.txt
jupyter lab case-02.ipynb
```

В Colab используется тот же файл. Начальная ячейка скачивает ZIP расследования, сверяет его SHA-256 и работает с вложенным `data/titanic.csv` без ручной правки путей.

## Файлы и происхождение

- `case-02.ipynb` — рабочая тетрадь;
- `solution/case-02-solution.ipynb` — эталон;
- `data/titanic.csv` — замороженный OpenML Titanic, dataset 40945, version 1;
- `data/SOURCE.md`, `data/LICENSE.txt`, `data/CHECKSUMS.sha256` — источник, лицензионная метка и контроль целостности;
- `check_result.md` — качественные и численные ориентиры.

Не заменяйте CSV свежей загрузкой: воспроизводимость аудита зависит от точной копии.

## Ритм семинара

Каждый этап тетради заканчивается наблюдаемым результатом и буквальной интерпретацией: источник и форма таблицы → схема и пропуски → временная доступность → единое разбиение → честный `Pipeline` → Dummy и основная модель → контролируемый антипример с утечкой → таблица ошибок → аудиторская записка.

Accuracy читается как доля правильных классов: больше лучше, но дисбаланс требует сравнения с `DummyClassifier`. ROC AUC оценивает ранжирование положительного класса по всем порогам: `0.5` соответствует случайному ранжированию, `1.0` — идеальному; эта метрика не равна accuracy и не гарантирует калибровку вероятностей.
